\begin{abstract}
\noindent
SlopGuard es una herramienta de línea de comandos que escanea las dependencias
Python de un proyecto y detecta paquetes inexistentes (alucinados por modelos de
lenguaje), \emph{typosquatting}, metadatos sospechosos o \textbf{malicia confirmada}
por inteligencia de amenazas comunitaria, \textbf{antes} de instalarlos. El
\textbf{Hito 1} entrega el motor determinista (Capas 0--2); el \textbf{Hito 2} añade
la \textbf{Capa 3 (\emph{threat-intel})}: consulta OSV.dev para detectar advisories
\texttt{MAL-*} y, opcionalmente, coteja contra la watchlist depscope de alucinaciones.
El diseño prioriza la \textbf{minimización de falsos positivos} como métrica de primera
clase y la \textbf{degradación segura}: ante un fallo de red, nunca reporta un falso
«todo bien». Cero dependencias de runtime (solo stdlib).

\medskip
\noindent\textbf{Resultados del Hito 1:} 619 pruebas en verde,
cobertura global 95{,}3\,\% ($\geq 90\,\%$), cobertura críticos 99\,\% ($\geq 95\,\%$);
\texttt{mypy --strict}; 2 contratos import-linter.

\medskip
\noindent\textbf{Resultados del Hito 2 (implementación final):} 1547 pruebas en verde,
cobertura global 96{,}23\,\% ($\geq 90\,\%$ requerido), cobertura en paquetes críticos
99\,\% ($\geq 95\,\%$ requerido, incluye \texttt{core/threatintel}); \texttt{mypy --strict}
limpio (82 archivos); 5 contratos de import-linter verificados; lint ruff (bandit S) sin
errores. Capa 3 verificada en vivo: \texttt{bioql} $\Rightarrow$ \texttt{MAL-2025-47868}
$\Rightarrow$ block. CI en GitHub Actions sobre Python 3.11 y 3.12.
\end{abstract}

\section{Introducción y problema}

Los asistentes de IA sugieren con frecuencia comandos \texttt{pip install} que
incluyen nombres de paquetes \emph{inexistentes}: alrededor del 20\,\% de los nombres
sugeridos no existen y una fracción significativa son \emph{near-misses} de paquetes
populares (USENIX Security 2025). Los atacantes pre-registran esos nombres alucinados
con código malicioso ---técnica conocida como \emph{slopsquatting}---, de modo que un
desarrollador que copia el comando sin verificar instala el paquete del atacante.

El único detector de código abierto previo es trivial: comprueba únicamente que PyPI
responda \texttt{200} para el nombre, sin análisis de similaridad ni de metadatos.
SlopGuard cubre ese hueco con un análisis multicapa, determinista y explicable.

\paragraph{Usuarios objetivo.}
\begin{itemize}[nosep]
  \item \textbf{Desarrollador Python con asistentes de IA}: verifica en segundos, sin
  fricción, que los paquetes sugeridos no sean alucinaciones ni \emph{typosquatting}.
  \item \textbf{Equipo DevSecOps}: un \emph{gate} de \emph{supply-chain} determinista en
  pre-commit/CI, con \emph{exit codes} estables y salida JSON, que falla de forma
  explícita y nunca da un falso «todo bien».
\end{itemize}

% ===========================================================================
\section{Alcance (Hitos 1 y 2)}

El producto completo tiene cinco capas (0--4) y varios \emph{frontends}. Los Hitos 1 y 2
construyen la arquitectura extensible y las cuatro capas implementadas:

\begin{itemize}[nosep]
  \item \textbf{Capa 0} --- existencia y edad del paquete vía PyPI JSON API.
  \item \textbf{Capa 1} --- \emph{typosquatting} por distancia de cadenas
  (Damerau--Levenshtein + Jaro--Winkler) contra el top-N de PyPI, sin red y determinista.
  \item \textbf{Capa 2} --- señales de metadatos (número de \emph{releases}, repositorio
  enlazado, completitud), con aporte acotado al \emph{score}.
  \item \textbf{Capa 3} --- \emph{threat-intel}: advisories \texttt{MAL-*} de OSV.dev
  (malicia confirmada) + watchlist depscope de alucinaciones conocidas (opcional).
\end{itemize}

\paragraph{Fuera de alcance (Hitos 1--2).} Análisis dinámico o ejecución del código de los
paquetes; Capa 4 (superficie de alucinación con LLM, cacheada); \emph{frontends} pre-commit
y GitHub Action como producto; soporte multi-ecosistema simultáneo (npm es post-MVP vía
adaptador); embeddings o modelos ML; escaneo de vulnerabilidades generales (CVE/GHSA).

\section{Arquitectura}

\subsection{Principios}
El sistema se divide en dos mitades con un contrato único:
\begin{itemize}[nosep]
  \item \textbf{Core} (\texttt{slopguard.core}): lógica pura y determinista, con
  \textbf{cero dependencias de runtime} (solo biblioteca estándar) y \textbf{cero}
  \emph{imports} de la CLI. Expone una API pública congelada.
  \item \textbf{Frontend} (\texttt{slopguard.cli}): \emph{argparse}, render
  humano/JSON, saneo de salida y mapeo a \emph{exit code}. No reimplementa reglas de
  dominio.
\end{itemize}

El motor de capas y \emph{scoring} nunca habla con PyPI: depende de la interfaz
\texttt{EcosystemAdapter} y del modelo normalizado \texttt{PackageMetadata}.
Añadir npm equivale a un adaptador nuevo, sin tocar capas ni \emph{scoring}.

  \item \texttt{core.layers.*}, \texttt{core.scoring.*} \;$\not\rightarrow$\;

  \item \texttt{core.threatintel.source} \;$\not\rightarrow$\; \texttt{core.net.*}
  (la interfaz/Protocol no conoce el transporte concreto --- R8.1).
  \item \texttt{core.layers.layer3\_threatintel} \;$\not\rightarrow$\;
  \texttt{core.threatintel.osv}, \texttt{core.threatintel.watchlist},
  \texttt{core.threatintel.composite}, \texttt{core.threatintel.resolver}
  (Capa 3 pura: solo consume \texttt{ThreatIntelResult} ya construido e inyectado
  por el engine --- R8.3).
\end{itemize}
Una regla de \emph{lint} (ruff/bandit) prohíbe \texttt{eval}/\texttt{exec}/
\texttt{pickle}/\texttt{marshal} sobre datos externos (NFR-Seg.2). Tests AST
verifican que el core no use \texttt{importlib} ni \texttt{__import__} sobre
nombres del manifiesto o de advisories OSV (NFR-Seg.1/Seg.3). El guardia estático
del allowlist de red verifica que \texttt{ALLOWED\_HOSTS} base $=\{$\texttt{pypi.org}$\}$
y que el conjunto efectivo por instancia $\subseteq \{$\texttt{api.osv.dev},
\texttt{depscope.dev}$\}$ (con \texttt{depscope.dev} solo si \texttt{enable\_watchlist}).
El modelo \texttt{Advisory} vive en \texttt{core.models} (módulo hoja) para que
\texttt{core.scoring.verdict} pueda importarlo sin cruzar la frontera con
\texttt{core.threatintel}.

\section{Modelo de resultado}

Para eliminar ambigüedad, el sistema distingue conceptos ortogonales:
\begin{itemize}[nosep]
  \item \textbf{score}: entero 0--100, solo para dependencias \emph{verificables}.
    Si el paquete no existe (404) el \emph{score} es \texttt{null}: el veredicto
    se asigna por \emph{override}, fuera de la función de \emph{scoring}.
  \item \textbf{verdict}: \texttt{allow} \textbar{} \texttt{warn} \textbar{}
  \texttt{block} (derivado del \emph{score} o por \emph{override}).
    Inexistencia (404) $\Rightarrow$ \texttt{verdict=block} con \texttt{score=null},
    independiente de \texttt{umbral_block}.
  \item \textbf{status}: \texttt{ok} (la verificación se completó) \textbar{}
  \texttt{unverifiable} (no se pudo verificar; sin \emph{score}, nunca
  \texttt{allow}).
  \item \textbf{error\_category}: \texttt{manifest_parse}, \texttt{invalid_config},
  \texttt{network_unverifiable}, \texttt{dataset_integrity}.
\end{itemize}

\paragraph{Clasificación de fallos de red.}
\begin{itemize}[nosep]
  \item \textbf{Transitorio} (reintentable; al agotar $\Rightarrow$ \texttt{unverifiable}):
    timeout de conexión/lectura, conexión caída/reset, \texttt{5xx}.
  \item \textbf{Permanente no reintentable:} \texttt{404} $\Rightarrow$
    \texttt{NOT_FOUND} (no es error). \texttt{403}, \texttt{410} y cualquier
    \texttt{4xx != 404} $\Rightarrow$ \texttt{unverifiable} (nunca
    \texttt{allow}).
  \item \textbf{Anomalía de seguridad:} redirección cross-scheme/cross-host,
    \texttt{Content-Length} excesivo, cuerpo $>$ \texttt{max_response_bytes},
    profundidad JSON $>$ \texttt{max_json_depth}, bomba de descompresión
    $\Rightarrow$ \texttt{NetworkUnverifiableError} $\Rightarrow$ \texttt{unverifiable}.
\end{itemize}

\paragraph{Exit codes (precedencia R7.5).}
\begin{center}
\small
\begin{tabular}{@{}cll@{}}
\toprule
\textbf{Código} & \textbf{Significado} & \textbf{Condición} \\
\midrule
0 & allow & todo \texttt{allow}, sin \emph{warn}/\emph{block}/\emph{unverifiable} \\
1 & warn & $\geq 1$ \emph{warn}, sin \emph{block} ni \emph{unverifiable} (sin \texttt{--strict}) \\
2 & block & $\geq 1$ \emph{block} (señal dominante); o cualquier \emph{warn} con \texttt{--strict} \\
3 & operacional & error total (manifiesto/config/dataset) \textbf{o} $\geq 1$ \emph{unverifiable} sin \emph{block} \\
\bottomrule
\end{tabular}
\end{center}

\noindent Precedencia: \texttt{block (2) > operacional/unverifiable (3) > warn (1) > allow (0)}.
Un \emph{block} confirmado domina sobre una verificación incompleta. Los \emph{unverifiable}
siempre se reportan en la salida aunque el exit code sea 2.

\paragraph{Algoritmo de agregación (función pura en \texttt{scoring/verdict.py}).}
\begin{lstlisting}[language=Python]
if error_operacional_total:    return 3  # manifest_parse/invalid_config/dataset_integrity
if any verdict == block:       return 2  # block domina
if any status == unverifiable: return 3
if any verdict == warn:        return 2 if strict else 1
return 0
\end{lstlisting}

\section{Capas de detección}

\paragraph{Capa 0 --- existencia y edad.} Si el paquete no existe en PyPI (404) se
emite la señal \texttt{NONEXISTENT} y el veredicto pasa a \texttt{block} por
\emph{override} (fuera de la función de \emph{scoring}), independiente de
\texttt{umbral_block}. Si existe pero su primera \emph{release} es más reciente que
\texttt{edad_minima_dias} días, se emite la señal \textbf{blanda}
\texttt{NEW_PACKAGE} (+15 puntos), que contribuye al \emph{score} pero
\textbf{nunca bloquea por sí sola} (invariante anti-FP). La edad se deriva de
\texttt{metadata.first_release_epoch} y de un \texttt{now_epoch}
\textbf{inyectado una vez por corrida} (determinismo reproducible en tests).

\paragraph{Capa 1 --- \emph{typosquatting}.} Se calcula la similaridad del nombre contra
el dataset embebido del top-N de PyPI usando Damerau--Levenshtein (DL) y
Jaro--Winkler (JW). Una entrada dispara señal si $1 \le \mathrm{DL} \le
\mathtt{dl\_max}$ \textbf{o} $\mathrm{JW} \ge \mathtt{jw\_min}$. El coste cuadrático se
acota con prefiltros sobre índices precomputados del \texttt{TopNDataset}: banda de
longitud $[L \pm \mathtt{dl\_max}]$ para DL; primer carácter para JW. Nombres de
$\le 3$ caracteres no emiten señal (R3.5); nombres más largos que
\texttt{nombre_max_chars} se marcan \texttt{NAME\_UNTRUSTED} sin ejecutar
distancias (anti-DoS cuadrático). El candidato objetivo se elige por menor DL $\to$
mayor JW $\to$ nombre ascendente (desempate determinista).

\paragraph{Capa 2 --- metadatos.} Solo fuentes de la PyPI JSON API normalizada. Se emite
\texttt{WEAK\_METADATA} (+7) si el paquete tiene $\le$ \texttt{releases\_min}
\emph{releases} \textbf{y} le faltan $\ge$ \texttt{metadata\_faltantes\_min} campos
del conjunto cerrado \{descripción, autor, licencia, clasificadores\}; y
\texttt{LOW\_VERIFIABILITY} (+5) si no enlaza repositorio. El aporte total de la
Capa 2 está \textbf{acotado} a \texttt{c2\_max\_contrib} puntos (cap duro). Cuando
un paquete tiene $\ge$ \texttt{releases\_populares} releases, repo enlazado y
metadatos completos, el cap se aplica igualmente, evitando falsos positivos por
popularidad baja relativa.

\paragraph{Capa 3 --- \emph{threat-intel} (Hito 2).}
La Capa 3 añade inteligencia de amenazas \emph{online} sobre las dependencias que
\emph{existen} en PyPI (estado \texttt{FOUND} de Capa 0). El engine introduce una
\textbf{barrera de lote} entre Capa 0 (concurrente por dependencia) y el bucle de
evaluación por dependencia: recolecta los nombres \texttt{FOUND}, los resuelve contra
la fuente de \emph{threat-intel} en lotes ($\le$ \texttt{osv\_batch\_max}), y luego
cada dependencia recibe su \texttt{ThreatIntelResult} ya construido.

\medskip
\noindent\textbf{Arquitectura \texttt{ThreatIntelSource}.} La interfaz \texttt{ThreatIntelSource}
(Protocol) es análoga a \texttt{EcosystemAdapter}: encapsula red, caché y parseo y
devuelve un modelo normalizado (\texttt{ThreatIntelResult}) por nombre. El motor de
capas/scoring \textbf{nunca} importa implementaciones de threat-intel (contratos
import-linter \S3 y \S5 de §3.3). \texttt{CompositeSource} hace fan-out a
\texttt{OsvSource} (siempre activa) y \texttt{WatchlistSource} (solo si
\texttt{enable\_watchlist}). Añadir una nueva fuente equivale a implementar el
Protocol sin tocar capas ni scoring.

\medskip
\noindent\textbf{Override \texttt{MALICIOUS} (ADR-06).} Un advisory \texttt{OSV} con
prefijo \texttt{MAL-} produce señal dura \texttt{MALICIOUS} (\texttt{is\_soft=False},
\texttt{weight=0}): \texttt{build\_dependency\_result} la detecta en la segunda rama
de precedencia, \textbf{antes} que \texttt{NONEXISTENT} y del scorer, y fija
\texttt{verdict=block}, \texttt{score=null}, poblando \texttt{advisories[]} con cada
ID validado. Las vulnerabilidades generales (\texttt{GHSA-*}, \texttt{CVE-*},
\texttt{PYSEC-*}) se ignoran. Verificado en vivo: \texttt{bioql} $\Rightarrow$
\texttt{MAL-2025-47868} $\Rightarrow$ \texttt{block}.

\medskip
\noindent\textbf{Señal \texttt{KNOWN\_HALLUCINATION} (ADR-07).} Match exacto en el corpus
depscope produce señal dura \texttt{KNOWN\_HALLUCINATION} con \texttt{weight=85}
($\ge$ \texttt{umbral\_block}=80): bloquea por \emph{score}, no por override, de modo
que el operador puede calibrar el umbral. La invariante anti-FP del Hito 1 (señales blandas
$\le 25 < \texttt{umbral\_warn}$) permanece intacta literalmente.

\medskip
\noindent\textbf{Degradación segura y precedencia de veredicto.}
Nuevo orden de 5 ramas en \texttt{build\_dependency\_result}:
\begin{enumerate}[nosep]
  \item \texttt{ctx.is\_unverifiable} (Capa 0) $\Rightarrow$ \texttt{status=unverifiable}.
  \item \texttt{MALICIOUS} $\Rightarrow$ \texttt{block}, \texttt{score=null}, \texttt{advisories[]}.
  \item \texttt{NONEXISTENT} $\Rightarrow$ \texttt{block}, \texttt{score=null}.
  \item Score normal (typosquat / \texttt{KNOWN\_HALLUCINATION}-85 / blandas) $\Rightarrow$ veredicto por umbral.
  \item \texttt{THREATINTEL\_UNVERIFIABLE} sin block/warn $\Rightarrow$ \texttt{unverifiable} o \texttt{warn}
        según \texttt{threatintel\_degraded\_status} (default \texttt{unverifiable}).
\end{enumerate}
Si OSV no responde, la dependencia pasa a \texttt{unverifiable} (exit 3), nunca a
\texttt{allow}. Un \texttt{block} de capas 0--2 domina sobre cualquier fallo de
threat-intel. \texttt{ctx.is\_unverifiable} queda ligado \textbf{exclusivamente} a Capa 0.

\medskip
\noindent\textbf{Privacidad y seguridad de la Capa 3.}
Solo se envía nombre normalizado (PEP 503) y ecosistema (\texttt{PyPI}) a
\texttt{api.osv.dev}; el allowlist de red se amplía por instancia (nunca la constante global).
El redirect handler valida contra el conjunto efectivo (fix SSRF). IDs de advisory validados
con \texttt{\^{}MAL-[0-9A-Za-z-]+\$}; URL reconstruida, nunca reflejada del payload. El
corpus depscope se valida por charset y cap al recibir \textbf{y} al leer caché
(anti-envenenamiento). \texttt{UNVERIFIABLE} nunca se persiste en caché.

\section{Función de \emph{scoring} (ADR-01)}

El \emph{score} es un modelo \textbf{aditivo con saturación} que separa señales
\textbf{duras} (basadas en el nombre; cruzan a \emph{warn}/\emph{block}) de
\textbf{blandas} (corroborantes, acotadas). Las señales TYPOSQUAT y NAME\_UNTRUSTED son
\textbf{mutuamente excluyentes}: si el nombre supera \texttt{nombre_max_chars}, se
emite NAME\_UNTRUSTED sin ejecutar los algoritmos de distancia.

\begin{center}
\small
\begin{tabular}{@{}llc@{}}
\toprule
\textbf{Clase} & \textbf{Señal / condición} & \textbf{Peso} \\
\midrule
Dura L3 & MALICIOUS (advisory \texttt{MAL-*} en OSV --- override, no scorer) & override \\
Dura L3 & KNOWN\_HALLUCINATION (match exacto corpus depscope, opcional) & 85 \\
Dura L1 & TYPOSQUAT (DL $=1$, mejor candidato top-N) & 60 \\
Dura L1 & TYPOSQUAT (DL $=2$) & 40 \\
Dura L1 & TYPOSQUAT (DL $>$ dl\_max \textbf{y} JW $\ge 0.95$) & 30 \\
Dura L1 & TYPOSQUAT (DL $>$ dl\_max \textbf{y} $0.92 \le$ JW $< 0.95$) & 25 \\
Dura L1 & NAME\_UNTRUSTED ($\mathtt{len} >$ \texttt{nombre_max_chars}) & 30 \\
Blanda L0 & NEW\_PACKAGE (primera release $<$ \texttt{edad_minima_dias} días) & $+15$ \\
Blanda L2 & WEAK\_METADATA (releases escasas y campos ausentes) & $+7$ \\
Blanda L2 & LOW\_VERIFIABILITY (sin repositorio enlazado) & $+5$ \\
Blanda L2 & \textit{Capa 2 total} (cap duro: min(WEAK+LOW, \texttt{c2_max_contrib})) & $\le 10$ \\
Blanda L3 & THREATINTEL\_UNVERIFIABLE (OSV/depscope no disponible, peso 0) & 0 \\
\bottomrule
\end{tabular}
\end{center}

\[
\mathtt{score} \;=\; \min\bigl(100,\; \mathtt{dura} + \min(\mathtt{blandas},\,25)\bigr)
\]

donde $\mathtt{blandas} = \mathtt{NEW\_PACKAGE} + \min(\mathtt{WEAK\_METADATA} +
\mathtt{LOW\_VERIFIABILITY},\; \mathtt{c2\_max\_contrib})$.

\paragraph{Invariante anti-FP (núcleo del diseño).} La suma máxima de señales blandas es
$15 + 10 = 25$, estrictamente \textbf{menor} que \texttt{umbral_warn} $= 50$. Por tanto
\emph{ninguna} combinación de señales blandas por sí sola puede producir \emph{warn} ni
\emph{block}: un paquete que \textbf{existe} y \textbf{no} dispara \emph{typosquat} nunca
supera 25 y siempre resulta \texttt{allow}. Esta propiedad se verifica por test de
invariante (sin señal dura ni inexistencia $\Rightarrow$ \texttt{score} $\le 49$).

\paragraph{Condición efectiva de \emph{block} ($\ge 80$).} Como las blandas suman $\le 25$,
alcanzar $\ge 80$ requiere una señal dura $\ge 55$, es decir \textbf{solo DL$=1$} (peso 60),
más $\ge 20$ de blandas. En palabras: \emph{bloqueo automático} $\iff$ nombre a una edición
de un paquete popular \textbf{y} recién publicado \textbf{y} con metadatos débiles. Esa
conjunción es la firma de un \emph{typosquat} real. Un paquete establecido con DL$=1$ (p.\,ej.
\texttt{attr} vs \texttt{attrs}) no es nuevo ni débil $\Rightarrow$ \emph{score} $\approx 60$
$\Rightarrow$ \texttt{warn} (humano confirma), nunca \emph{block}.

\paragraph{Casos de \emph{score} nulo.} Inexistencia (404): \texttt{verdict=block},
\texttt{score=null} (\emph{override}, independiente de \texttt{umbral_block}).
\texttt{unverifiable}: \texttt{status=unverifiable}, \texttt{score=null},
\texttt{verdict=null}, nunca \texttt{allow}.

\section{Propiedades de seguridad verificadas}

SlopGuard es una herramienta de seguridad; su propia superficie de ataque importa.
Las propiedades siguientes se verifican por análisis estático (lint, import-linter,
tests AST) o por tests de comportamiento con servidor local malicioso (T15):

\begin{itemize}[nosep]
  \item \textbf{Nunca ejecuta código de los paquetes analizados}: solo inspecciona
  metadatos. Sin \texttt{eval}/\texttt{exec}/\texttt{pickle}/\texttt{marshal}
  sobre datos externos (ruff/bandit, NFR-Seg.2). Sin \texttt{importlib} ni
  \texttt{__import__} sobre nombres del manifiesto (test AST, NFR-Seg.1).
  \item \textbf{HTTP endurecido (ADR-03 + Hito 2)}: solo HTTPS con verificación de certificado
  activa (no desactivable), \emph{allowlist} de host base \texttt{\{pypi.org\}} ampliada por
  instancia con \texttt{extra\_allowed\_hosts} (\texttt{api.osv.dev} siempre si L3 activa;
  \texttt{depscope.dev} solo si watchlist). El redirect handler valida contra el conjunto
  efectivo de la instancia (fix SSRF: previene redirecciones cross-host desde
  \texttt{api.osv.dev}). Lectura \emph{streaming} acotada por \texttt{max\_response\_bytes},
  mitigación de bombas de descompresión, \texttt{safe\_json\_loads} (\texttt{max\_json\_depth}).
  Anomalías $\Rightarrow$ \texttt{NetworkUnverifiableError} $\Rightarrow$
  \texttt{unverifiable} (nunca \texttt{allow}).
  \item \textbf{Caché segura (ADR-05)}: JSON exclusivamente (nunca \emph{pickle}), escritura
  atómica (\texttt{tempfile} + \texttt{os.replace}), permisos 0700/0600, claves
  \texttt{sha256(ecosystem:name)} (anti \emph{path traversal} por construcción),
  validación defensiva completa al leer (esquema, tipos, rangos, TTL $\Rightarrow$ miss
  nunca crashear). \texttt{unverifiable} no se persiste (no cachear fallos transitorios).
  \item \textbf{Integridad del dataset (ADR-01/NFR-Seg.7)}: el top-N embebido lleva
  \texttt{version}, \texttt{generated_at} y procedencia documentada; se verifica
  con SHA-256 al cargar. Dataset ausente/corrupto $\Rightarrow$
  \texttt{DatasetIntegrityError} (\texttt{exit 3}), nunca Capa 1 limpia en silencio.
  \item \textbf{Anti-inyección de salida (R6.5)}: todo nombre o dato externo se sanea
  (ANSI CSI/SGR, controles C0/C1, CR/LF) en salida humana, logs y JSON. Sin rutas absolutas
  del sistema ni contenido del manifiesto en mensajes de error de CI.
  \item \textbf{Privacidad y costo cero}: nunca se envía el manifiesto a terceros; las
  capas 0--2 consultan PyPI \emph{solo por nombre de paquete}; la Capa 3 envía solo
  nombre normalizado + ecosistema a \texttt{api.osv.dev} (y opcionalmente
  \texttt{depscope.dev}); desactivable con \texttt{--no-layer3}; sin LLM ni servicios
  de pago; cero dependencias de runtime.
  \item \textbf{Anti-envenenamiento de feed (Hito 2)}: IDs de advisory OSV validados con
  \texttt{\^{}MAL-[0-9A-Za-z-]+\$} antes de construir URLs; corpus depscope validado por
  charset \texttt{\^{}[a-z0-9-]+\$} y cap de tamaño al parsear \textbf{y} al leer caché;
  corpus inflado $\Rightarrow$ miss (refetch), no truncamiento silencioso.
\end{itemize}

\section{Manual de uso (CLI)}

\begin{lstlisting}[language=]
# Escanear un manifiesto (Capa 3 activa por defecto)
slopguard scan requirements.txt
slopguard scan pyproject.toml --strict
pip freeze | slopguard scan -            # lee de stdin (formato freeze)

# Solo capas deterministas (sin red a OSV ni depscope)
slopguard scan requirements.txt --no-layer3

# Activar watchlist de alucinaciones depscope (opt-in)
slopguard scan requirements.txt --enable-watchlist

# Salida apta para CI (JSON 1.1, versionado)
slopguard scan requirements.txt --format json

# Forzar tipo de manifiesto; ajustar umbrales; deshabilitar cache
slopguard scan deps.txt --manifest-type requirements --umbral-block 75 --no-cache
\end{lstlisting}

\noindent\textbf{Flags principales:} \texttt{--format \{human,json\}},
\texttt{--no-cache}, \texttt{--strict}, \texttt{--config <ruta>},
\texttt{--ecosystem pypi}, \texttt{--manifest-type \{requirements,pyproject,freeze\}},
\texttt{--no-layer3} (desactiva Capa 3), \texttt{--enable-watchlist} (activa depscope),
y \emph{overrides} de umbrales/red. Precedencia: CLI $>$ archivo de config $>$
\emph{defaults}.

\paragraph{Escenarios representativos.}

\begin{itemize}[nosep]
  \item \texttt{reqursts} (typosquat de \texttt{requests}, DL$=1$)
    $\Rightarrow$ \texttt{verdict=block}, \texttt{score=75+} si también es nuevo,
    \texttt{suspected_target="requests"}.
  \item Paquete con nombre inventado / alucinado $\Rightarrow$ PyPI 404
    $\Rightarrow$ \texttt{verdict=block}, \texttt{score=null} (override).
  \item \texttt{boto3}, \texttt{requests}, \texttt{django} (populares, existentes)
    $\Rightarrow$ señales blandas mínimas $\Rightarrow$ \texttt{verdict=allow}.
  \item PyPI inaccesible $\Rightarrow$ \texttt{status=unverifiable}, \texttt{exit 3},
    nunca falso «todo bien».
\end{itemize}

\paragraph{Ejemplo de salida JSON (\texttt{schema\_version} 1.0).}
\begin{lstlisting}[language=]
{
  "schema_version": "1.0", "tool_version": "0.1.0", "ecosystem": "pypi",
  "summary": {"total":3,"allow":1,"warn":0,"block":2,
              "unverifiable":0,"exit_code":2},
  "error_category": null,
  "results": [
    {"name":"paquete-inexistente-xyz","version_pin":null,
     "status":"ok","verdict":"block","score":null,
     "suspected_target":null,"error_category":null,
     "signals":[{"layer":0,"code":"nonexistent","weight":0,
       "is_soft":false,
       "detail":"El paquete no existe en PyPI (404).","suspected_target":null}]},
    {"name":"reqursts","version_pin":null,
     "status":"ok","verdict":"block","score":75,
     "suspected_target":"requests","error_category":null,
     "signals":[
       {"layer":1,"code":"typosquat","weight":60,"is_soft":false,
        "detail":"El nombre se parece a 'requests' (distancia 1).",
        "suspected_target":"requests"},
       {"layer":0,"code":"new_package","weight":15,"is_soft":true,
        "detail":"Publicado hace 4 dias (umbral 90).","suspected_target":null}]},
    {"name":"boto3","version_pin":null,
     "status":"ok","verdict":"allow","score":3,
     "suspected_target":null,"error_category":null,"signals":[]}
  ]
}
\end{lstlisting}

La salida JSON es estable, versionada y \textbf{sin marcas de tiempo} (determinismo
garantizado para CI). El orden es determinista: primero \texttt{unverifiable}, luego
\texttt{block}, luego \texttt{warn}, luego \texttt{allow}; a igual veredicto,
por nombre normalizado ascendente.

\paragraph{Uso como \emph{gate} de CI (GitHub Actions).}
\begin{lstlisting}[language=]
- name: SlopGuard -- supply chain gate
  run: slopguard scan requirements.txt --strict --format json
  # exit 0: todo allow. exit 2: block (o warn con --strict).
  # exit 3: error operacional o paquete unverifiable.
\end{lstlisting}

\paragraph{Uso como \emph{hook} de pre-commit.}
\begin{lstlisting}[language=]
# .pre-commit-config.yaml
repos:
  - repo: local
    hooks:
      - id: slopguard
        name: SlopGuard supply-chain guard
        entry: slopguard scan
        args: [requirements.txt, --strict]
        language: python
        pass_filenames: false
        always_run: true
\end{lstlisting}

\subsection{Pipeline de CI/CD}

El workflow \texttt{.github/workflows/slopguard-ci.yml} ejecuta en GitHub Actions
sobre una matriz Python 3.11/3.12:

\begin{enumerate}[nosep]
  \item \textbf{Ruff} — lint + reglas de seguridad (bandit).
  \item \textbf{Mypy} — tipado estricto (\texttt{--strict}).
  \item \textbf{Import-linter} — contratos de arquitectura R10.
  \item \textbf{Pytest + cobertura global} — gate $\ge 90\,\%$ (\texttt{--cov-fail-under=90}).
  \item \textbf{Cobertura de paquetes críticos} — gate $\ge 95\,\%$ sobre los 5 paquetes.
  \item \textbf{Compilación LaTeX} — este documento a PDF (\texttt{xelatex}) como artefacto.
\end{enumerate}

El build falla si cualquier gate no se cumple. Los umbrales de cobertura se declaran en
\texttt{pyproject.toml} (\texttt{[tool.coverage.report].fail_under}) y en los
pasos del CI, como única fuente de verdad.
