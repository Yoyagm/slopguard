# Plan de Tareas: SlopGuard (Hito 2 — Capa 3 Threat-Intel)

> Fase 3 del flujo Spec-Driven. Fuentes de verdad APROBADAS: `requirements.md` (R1-R8 + NFR en EARS) y `design.md` + `design-parte2.md` (arquitectura §1, modelos §2, contratos §3, flujo §4, ADRs 06-10 §5, trazabilidad §6, alto riesgo §8). Tareas discretas, pequeñas, verificables, ordenadas por dependencias (ningún ID depende de un ID posterior). Base: Hito 1 completo y verde (619 tests).

## Convenciones
- **developer**: implementación estándar (lógica determinista, parsers, glue, config, render). Tipado estricto, funciones ≤50 líneas, docstrings ES, cero deps runtime.
- **developer-complex**: SOLO las 4 áreas de alto riesgo del design §8 — (1) transporte a hosts nuevos `post_json`+allowlist por-instancia+redirect handler (RISK-H2-1), (2) parseo defensivo OSV/depscope + caché threat-intel (RISK-H2-2), (3) intercalado batch+concurrencia en el engine (RISK-H2-3), (4) override de veredicto MALICIOUS + precedencias + degradación (RISK-H2-4). Toda tarea developer-complex **requiere security-reviewer**.
- **tester**: pruebas (unitarias, tabla, propiedad, red con servidor local malicioso, e2e), gates de cobertura, propiedades estructurales (import-linter, AST, guardia de allowlist).
- **critic**: Compuerta 3 final, APROBAR/RECHAZAR contra criterios EARS y propiedades estructurales.
- **documenter**: README/CHANGELOG/LaTeX + persistencia en Basic Memory (proyecto "equipo", [decision]).

**Definición de done por tarea:** code-reviewer → security-reviewer (si developer-complex o toca red/caché/dataset/saneo/propiedad de seguridad) → tester verde → critic.

**Piso de cobertura (gate):** líneas+ramas **≥90% global** y **≥95%** en paquetes críticos: `core/net`, `core/cache`, `core/scoring`, `core/layers`, `core/dataset`, **`core/threatintel` (nuevo)**.

**Compatibilidad:** cambios aditivos; Hito 1 (capas deterministas, exit codes, JSON 1.0) intacto; `schema_version` 1.0→1.1; caché threat-intel con `cache_schema_version="ti-1"` separado.
